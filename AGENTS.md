# Repository Guidelines

## Project Structure & Module Organization
- `lib/` contains the gem implementation. Core WeChat APIs, tokens, tickets, messages, responders, and helpers live under `lib/wechat/`; Rails responder integration is in `lib/action_controller/`; install/config/menu generators and templates are in `lib/generators/wechat/`.
- `spec/` contains RSpec tests. `spec/lib/wechat/*_spec.rb` covers gem behavior, `spec/support/` holds shared helpers, and `spec/dummy/` is the Rails test app.
- `bin/wechat` is the executable, `certs/` stores gem signing certificates, `pkg/` holds built gems, and `coverage/` is generated by SimpleCov.
- User-facing documentation is in `README.md`, `README-CN.md`, and `CHANGELOG.md`.

## Build, Test, and Development Commands
- `bundle install` installs the gem dependencies from `Gemfile` and `wechat.gemspec`.
- `bundle exec rake` runs the default verification task: RSpec followed by RuboCop.
- `bundle exec rspec` runs the full test suite; use `bundle exec rspec spec/lib/wechat/api_spec.rb` for a focused file.
- `bundle exec rubocop` lints production Ruby files under `lib/`.
- `bundle exec rake build` builds the gem into `pkg/`; run it before release-related changes.

## Coding Style & Naming Conventions
- Use Ruby conventions: 2-space indentation, `snake_case` methods/files, and `CamelCase` classes/modules.
- Keep production code compatible with the configured RuboCop target (`TargetRubyVersion: 2.6`) unless the compatibility policy is intentionally updated.
- Prefer `# frozen_string_literal: true`, single-quoted strings when interpolation is not needed, and concise methods.
- RuboCop allows 180-character lines and excludes specs; keep `lib/` lint-clean and avoid inline disables unless the reason is explicit.

## Testing Guidelines
- Add or update RSpec examples in matching `spec/.../*_spec.rb` files.
- Tests load `spec/dummy`, use an in-memory sqlite schema, and set default WeChat environment variables in `spec/spec_helper.rb`.
- Avoid real network access in tests; mock HTTP/API boundaries and keep examples deterministic.
- SimpleCov starts automatically from `spec/spec_helper.rb`; do not commit generated `coverage/` changes.

## Commit & Pull Request Guidelines
- Keep commits focused with concise, imperative subjects. Existing history uses both plain subjects (`Add agentid`) and conventional prefixes (`chore: ...`, `ci(semaphore): ...`).
- For pull requests, include a clear summary, test results such as `bundle exec rake`, linked issues when relevant, and screenshots or generated config snippets for generator-facing changes.
- Update README examples and `CHANGELOG.md` when behavior, public APIs, or configuration templates change.
